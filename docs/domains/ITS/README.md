### Итерация 1: Определение контекста

**IT-инженер**: Ключевое ограничение — сельская местность. В центральной России за пределами 30 км от областных центров мобильное покрытие 4G крайне неравномерное. Нужно заложить три уровня связи: внешний канал в интернет, внутренняя поселковая сеть, оперативная радиосвязь. Каждый уровень должен работать автономно.

**Разработчик ПО**: Информационная система поселения — это не «сайт». Это ERP-система для управления общиной: база жителей, учёт домов и участков, расписание работ, учёт продукции (с/х, производство), финансовый учёт, склады. Система должна работать локально на сервере поселения, без зависимости от облака.

**Специалист по связи**: Для оперативной координации (строительство, с/х, дежурства, ЧС) нужна радиосвязь. Мобильный телефон ненадёжен — зависит от оператора, от покрытия, от электричества на вышке. Рации работают автономно и покрывают территорию поселения.

**Специалист по документообороту**: Поселение — это инженерный проект с сотнями документов: чертежи, расчёты, договоры, протоколы собраний, технические регламенты. Без системы всё потеряется через год. Нужен электронный архив с первого дня.

**Специалист по кибербезопасности**: Критический принцип — **offline-first**. Поселение должно функционировать при полном отключении интернета. Все данные хранятся локально. Интернет — удобство, не зависимость. Резервные копии — на физические носители с хранением за пределами поселения.

### Итерация 2: Каналы связи с внешним миром

**IT-инженер**: Три реалистичных варианта интернет-доступа для сельской местности в России 2025-2026:

1. **Starlink** — спутниковый интернет. Скорость 50–250 Мбит/с, пинг 20–40 мс. Комплект оборудования V3.0 около 65 000 руб. Абонентская плата ~120$/мес (около 12 000 руб/мес по текущему курсу). Работает в России, но официально не поддерживается — оплата через посредников. Юридически серая зона.

2. **Сотовый 4G LTE** — через внешнюю направленную антенну и усилитель сигнала. Реалистичная скорость 10–50 Мбит/с. Зависит от расстояния до вышки. Комплект (антенна + модем + усилитель) — 15 000–40 000 руб. Абонентская плата — безлимитные тарифы 500–1500 руб/мес (МТС, Билайн, Мегафон, Yota, Т-мобайл).

3. **Оптоволокно (ВОЛС)** — если рядом проходит магистраль. Самый надёжный и быстрый вариант. Подведение стоит от 50 000 до 500 000+ руб в зависимости от расстояния. Абонентская плата 500–3000 руб/мес. Реалистично только если до ближайшего узла не более 5–10 км.

**Специалист по кибербезопасности**: Рекомендую комбинировать минимум два канала. Основной — 4G LTE (дешевле, легальнее). Резервный — Starlink (для критических задач, если 4G отказывает). В идеале оба одновременно с автоматическим переключением.

### Итерация 3: Российские спутниковые альтернативы

**IT-инженер**: Нужно учитывать российские проекты спутникового интернета:

- **«Бюро 1440» (ИКС-Холдинг)** — разрабатывает систему «Рассвет». Терминал для физлиц ~50 000 руб, абонентская плата ~8 000 руб/мес. Запуск спутников перенесён с 2025 на 2026 год. Пока не работает.
- **«Сфера» (Роскосмос)** — проект федерального масштаба на 445 млрд руб. Долгосрочная перспектива.
- **«Скиф» и «Экспресс-РВ»** — на этапе разработки.

Вывод: на 2025-2026 год реально доступны только Starlink (серая зона) и 4G LTE.

**Специалист по связи**: Для начального этапа (строительный лагерь, 10–20 человек) хватит одного 4G-канала. Starlink имеет смысл подключать при заселении первых семей, когда появляется потребность в стабильном интернете для школы и удалённой работы.

### Итерация 4: Усиление сотового сигнала

**Специалист по связи**: В сельской местности сотовый сигнал часто слабый. Решение:

1. **Направленная антенна** на мачте 6–12 м, нацеленная на ближайшую вышку оператора. Антенна типа MIMO 4G LTE (например, Антэкс AGATA, КАА-15) — усиление 15–20 dBi. Цена: 3 000–8 000 руб.

2. **4G-модем/роутер** с внешними антенными разъёмами. Huawei B315/B525 или аналоги. Цена: 5 000–15 000 руб.

3. **Репитер (усилитель сигнала)** для покрытия нескольких зданий. Мультидиапазонный GSM/3G/4G репитер (900/1800/2100/2600 МГц). Цена комплекта с антеннами: 15 000–50 000 руб.

**IT-инженер**: Важный момент — определить оператора с наилучшим покрытием в конкретном районе ДО покупки земли. Карты покрытия публикуют МТС, Билайн, Мегафон, Tele2 на своих сайтах. Рекомендую проверять на месте тестовым замером.

### Итерация 5: Внутренняя поселковая сеть

**IT-инженер**: Для поселения на 200–500 домов нужна полноценная локальная сеть. Архитектура:

- **Центральный узел**: серверная комната в административном здании. Стойка с сервером, коммутатором, роутером, ИБП.
- **Магистраль**: оптоволоконный кабель (одномод, FTTH) по основным улицам. Альтернатива для начала — витая пара Cat6 (до 100 м между точками).
- **Точки доступа Wi-Fi**: на столбах или зданиях каждые 50–100 м. Уличные, всепогодные.
- **Домовые подключения**: Ethernet-кабель от ближайшей точки доступа до дома. Или Wi-Fi для удалённых домов.

**Разработчик ПО**: Внутренняя сеть необходима не только для интернета. Она обеспечивает:
- Доступ к информационной системе поселения (веб-интерфейс)
- Внутреннюю телефонию (VoIP)
- Видеонаблюдение (по необходимости)
- Общий файловый сервер (документация, чертежи)
- Внутренний мессенджер

### Итерация 7: Радиосвязь — стандарт и частоты

**Специалист по связи**: Для оперативной связи внутри поселения оптимальный стандарт — **DMR (Digital Mobile Radio)**. Преимущества:
- Цифровое качество звука (нет шумов аналоговой связи)
- Два разговора одновременно на одной частоте (TDMA — 2 тайм-слота)
- Экономия батареи (до 40% дольше аналоговых раций)
- Шифрование связи
- Текстовые сообщения, GPS-позиционирование
- Совместимость между производителями (открытый стандарт ETSI)

**Частотный диапазон**: UHF 400–470 МГц — оптимален для сельской местности с лесами и холмами. Сигнал лучше проходит через растительность, чем VHF.

**Специалист по кибербезопасности**: DMR поддерживает шифрование AES-256. Для поселковой связи это важно — оперативные переговоры о расписании, координация работ, ЧС не должны быть доступны посторонним.

### Итерация 10: Информационная система — функциональные требования

**Разработчик ПО**: Информационная система поселения (ИСП) — центральное ПО. Модули:

**1. Реестр жителей**
- ФИО, дата рождения, контакты, дом/участок
- Семьи, дети (для школы, медпункта)
- Навыки и специальности
- Трудовой вклад (учёт отработанных часов по направлениям)
- Статус (постоянный житель, кандидат, временный)

**2. Управление территорией**
- Реестр домов и участков (адрес, площадь, состояние)
- Карта поселения с объектами
- Учёт инженерных сетей (водопровод, электрика, канализация)
- Журнал обслуживания инфраструктуры

**3. Планирование и задачи**
- Расписание работ (кто, где, когда)
- Дежурства (охрана, пожарная, дежурный по коровнику)
- Задачи по подразделениям (строительство, с/х, производство)
- Канбан-доска для текущих проектов

**4. Сельскохозяйственный учёт**
- Карта полей и теплиц (что где посажено, стадия роста)
- Поголовье животных (вид, возраст, продуктивность, вакцинация)
- Урожай (сбор, хранение, распределение)
- Запасы семян и кормов

**5. Производственный учёт**
- Выпуск продукции (домокомплекты, пиломатериал, бетон)
- Складской учёт (материалы, инструмент, запчасти)
- Заказы и логистика

**6. Финансовый учёт**
- Бюджет поселения (доходы/расходы)
- Учёт взносов жителей
- Расчёты с поставщиками и покупателями
- Учёт трудочасов и распределение

**7. Документооборот**
- Хранилище документов (чертежи, регламенты, договоры)
- Протоколы собраний
- Входящая/исходящая корреспонденция
- Шаблоны документов

**8. Коммуникации**
- Объявления и новости поселения
- Внутренний мессенджер
- Доска объявлений
- Голосования по решениям общины

### Итерация 11: Архитектура ПО — технический стек

**Разработчик ПО**: Ключевые требования к архитектуре:
- **Локальная установка** (self-hosted) — никакой зависимости от облака
- **Веб-интерфейс** — доступ с любого устройства (ПК, планшет, смартфон)
- **Офлайн-совместимость** — критические функции работают без интернета
- **Низкие требования к железу** — должно крутиться на 1 сервере

**Вариант 1: Собственная разработка** (расширение существующего SLOBODA)
- Уже есть React + Vite + TypeScript (client/) + Express.js + PostgreSQL (server/)
- Нужно расширить модулями ИСП (6–12 мес разработки при 1-2 разработчиках)
- Плюс: полный контроль, точное соответствие нуждам
- Минус: долго, дорого в разработке

**Вариант 2: Комбинация готовых Open Source решений**
- **ERPNext** (frappe/erpnext) — полноценная ERP: бухгалтерия, HR, складской учёт, производство, сельское хозяйство. Python/JavaScript. Self-hosted. Бесплатно.
- **Nextcloud** — файловый сервер, совместная работа с документами, календари, контакты. Аналог Google Workspace, но локальный. Бесплатно.
- **Rocket.Chat** или **Mattermost** — корпоративный мессенджер. Self-hosted. Бесплатно.
- Связка через API или n8n (автоматизация рабочих процессов).

**Вариант 3: Гибрид**
- ERPNext для учёта и производства
- Nextcloud для документов и файлов
- Собственные модули для уникальных задач (реестр жителей, карта поселения, трудовой учёт)
- Единый портал с single sign-on (SSO)

**Специалист по кибербезопасности**: Рекомендую Вариант 3. ERPNext и Nextcloud — зрелые проекты с активным сообществом и обновлениями безопасности. Собственный код только для того, чего нет в готовых решениях.

### Итерация 12: Документооборот — практика

**Специалист по документообороту**: Типы документов поселения:

**Организационно-правовые:**
- Устав поселения / кооператива
- Правила внутреннего распорядка
- Положения о подразделениях
- Протоколы общих собраний
- Решения совета

**Договоры и юридические:**
- Договоры с поставщиками материалов
- Договоры продажи продукции
- Трудовые/волонтёрские соглашения
- Земельные документы (кадастровые паспорта, выписки ЕГРН)

**Техническая документация:**
- Генеральный план поселения
- Проекты домов (чертежи, расчёты, BOM)
- Проекты инженерных сетей
- Паспорта оборудования и станков
- Руководства по эксплуатации

**Бухгалтерские:**
- Бюджет и финансовый план
- Акты, накладные, счета
- Инвентаризационные ведомости

**Регламенты и инструкции:**
- Технологические карты (производство домокомплектов, сыроварение, и т.д.)
- Инструкции по технике безопасности

### Итерация 17: Кибербезопасность

**Специалист по кибербезопасности**: Угрозы и защита для поселения:

**1. Потеря данных:**
- RAID на серверах — защита от отказа дисков
- Ежедневное резервное копирование на NAS
- Еженедельное копирование на внешний HDD (хранится за пределами серверной)
- Ежемесячное копирование на внешний HDD, который увозится за пределы поселения (защита от пожара)

**2. Несанкционированный доступ:**
- Все учётные записи с надёжными паролями (минимум 12 символов)
- Разграничение прав (не все видят финансы, не все редактируют документы)
- Авторизация через единый сервер (LDAP / Keycloak)
- Шифрование дисков сервера (LUKS)
- Физическая безопасность серверной (замок, журнал доступа)

**3. Вирусы и вредоносное ПО:**
- Серверы на Linux — минимальная поверхность атаки
- Рабочие станции на Linux
- Регулярные обновления ОС и ПО
- Firewall на маршрутизаторе (MikroTik — встроенный firewall)

**4. Отключение электричества:**
- ИБП для серверов (15–30 мин)
- Автоматическое корректное выключение при разряде батареи
- Генератор как резервный источник

**5. Отказ оборудования:**
- RAID-зеркалирование для дисков
- Запасной мини-ПК (заранее настроенный) для быстрого восстановления сервера
- Инструкция по восстановлению из бэкапа (должна быть распечатана на бумаге!)

### Итерация 21: Mesh-сеть для сплошного покрытия поселения

**Сетевой инженер**: Классическая звездообразная топология (все точки доступа подключены кабелем к центральному коммутатору) имеет критический недостаток — при обрыве одного кабеля целый квартал теряет связь. Для поселения с протяжённой территорией в ~10 га нужна **mesh-сеть** — самоорганизующаяся ячеистая сеть, в которой каждый узел может ретранслировать трафик соседям.

**Архитектура mesh-сети поселения:**

```
                     ИНТЕРНЕТ
                        │
                   ┌────┴────┐
                   │ Gateway │ ← MikroTik hAP ax³ (центральный шлюз)
                   └────┬────┘
                        │ Ethernet (магистраль)
           ┌────────────┼────────────┐
           │            │            │
     ┌─────┴─────┐ ┌───┴───┐ ┌─────┴─────┐
     │  Mesh #1  │ │Mesh #2│ │  Mesh #3  │  ← Опорные узлы (на столбах/крышах)
     │  Адм.     │ │ Склад │ │  Школа    │     MikroTik Audience LTE6
     └─────┬─────┘ └───┬───┘ └─────┬─────┘
           │   ╲       │      ╱    │
           │    ╲      │     ╱     │           ← Mesh-линки (802.11s)
           │     ╲     │    ╱      │              автоматическая маршрутизация
     ┌─────┴─────┐ ┌───┴───┐ ┌────┴──────┐
     │  Mesh #4  │ │Mesh #5│ │  Mesh #6  │  ← Узлы жилых кварталов
     │  Квартал 1│ │Кварт.2│ │  Ферма    │     MikroTik cAP ax
     └───────────┘ └───────┘ └───────────┘
```

**Сетевой инженер**: Конкретное решение — **MikroTik с поддержкой 802.11ax (Wi-Fi 6)**:

| Компонент | Модель | Назначение | Цена (руб) |
|-----------|--------|-----------|------------|
| Центральный шлюз | MikroTik hAP ax³ | Gateway, CAPsMAN-контроллер | 12 000 |
| Опорные mesh-узлы | MikroTik Audience LTE6 kit | Уличные mesh-точки с LTE fallback | 18 000 × 6 = 108 000 |
| Квартальные mesh-узлы | MikroTik cAP ax | Внутри/снаружи зданий | 8 500 × 12 = 102 000 |
| Дальние mesh-узлы | MikroTik SXTsq Lite5 | Линк до фермы/теплиц (1–3 км) | 5 500 × 4 = 22 000 |
| **ИТОГО mesh-сеть** | | | **244 000** |

**Системный администратор**: MikroTik CAPsMAN v2 позволяет управлять всеми точками доступа централизованно — единое имя сети (SSID), бесшовный роуминг, автоматическое переключение клиентов на ближайший узел. Настройка через WinBox или веб-интерфейс. Обновление прошивки — одной командой на все устройства.

**Специалист по информационной безопасности**: Mesh-сеть обязательно должна иметь WPA3-SAE шифрование. Для устройств без поддержки WPA3 — отдельный SSID с WPA2-AES и ограниченным доступом (только интернет, без доступа к серверам). Гостевая сеть — изолированная VLAN без доступа к внутренним ресурсам.

**Преимущества mesh для поселения:**
- Самовосстановление: при выходе одного узла трафик перенаправляется через соседей
- Масштабируемость: новый дом = новый mesh-узел, подключается автоматически
- Нет единой точки отказа (кроме gateway, который дублируется)
- Бесшовный роуминг: человек идёт по поселению, устройство автоматически переключается

### Итерация 23: Open-source ERP — ERPNext vs Odoo для управления поселением

| Параметр | ERPNext | Odoo Community |

### Итерация 24: IoT-сенсоры — мониторинг окружающей среды и инфраструктуры

**Сетевой инженер**: IoT (Internet of Things) — датчики, подключённые к локальной сети и передающие данные автоматически. Для поселения это не «умный дом» — это **производственная необходимость**: мониторинг погоды, состояния грунта, уровня воды, потребления энергии.

**Протокол связи для IoT:**
Для поселения оптимальна комбинация:
- **LoRa/LoRaWAN** (Long Range) — дальность до 5–15 км, низкое энергопотребление, работа от батареи до 2 лет. Идеально для датчиков на полях, теплицах, водозаборе.
- **Zigbee** — для плотных сетей в зданиях (теплицы, склады, серверная).
- **Wi-Fi** — для устройств с постоянным питанием (камеры, энергомониторы).

**Архитектура IoT поселения:**

```
    Датчики (LoRa)                    Датчики (Zigbee)
   ╔════════════════╗               ╔═══════════════╗
   ║ Метеостанция   ║               ║ Теплица #1    ║
   ║ Влажность почвы║               ║ Теплица #2    ║
   ║ Уровень воды   ║               ║ Серверная     ║
   ║ Солнечные пан. ║               ║ Склад         ║
   ╚═══════╤════════╝               ╚══════╤════════╝
           │ LoRa (868 МГц)                │ Zigbee (2.4 ГГц)
           │                               │
    ┌──────┴──────┐                 ┌──────┴──────┐
    │ LoRa Gateway│                 │Zigbee Coord.│
    │ RAK7268 V2  │                 │ Sonoff ZB   │
    └──────┬──────┘                 └──────┬──────┘
           │ Ethernet                      │ USB
           └──────────────┬────────────────┘
                          │
                   ┌──────┴──────┐
                   │   Сервер    │
                   │ ChirpStack  │  ← LoRaWAN Network Server
                   │ Zigbee2MQTT │  ← Мост Zigbee → MQTT
                   │ Mosquitto   │  ← MQTT-брокер
                   │ Home Assist.│  ← Дашборд и автоматизация
                   │ InfluxDB    │  ← Хранение временных рядов
                   │ Grafana     │  ← Визуализация графиков
                   └─────────────┘
```

**Конкретные датчики и оборудование:**

| Категория | Датчик/Устройство | Модель | Протокол | Цена (руб) |
|-----------|-------------------|--------|---------|------------|
| **Метеостанция** | Температура + влажность + давление + ветер + осадки | Ecowitt WS90 + GW2000 | Wi-Fi | 25 000 |
| **Влажность почвы** | Ёмкостный датчик (3 шт по полям) | Dragino LSE01 | LoRa | 3 × 8 000 = 24 000 |
| **Уровень воды** (скважина/ёмкость) | Ультразвуковой уровнемер | Dragino LDDS75 | LoRa | 2 × 9 000 = 18 000 |
| **Температура теплиц** | Цифровой термогигрометр (6 шт) | Sonoff SNZB-02D | Zigbee | 6 × 1 200 = 7 200 |
| **Открытие дверей** (склад, серверная) | Контактный датчик | Sonoff SNZB-04 | Zigbee | 5 × 800 = 4 000 |
| **Энергомониторинг** | Счётчик на DIN-рейку + CT-трансформатор | Shelly Pro 3EM | Wi-Fi | 3 × 12 000 = 36 000 |
| **LoRa Gateway** | Шлюз LoRaWAN (868 МГц, 8 каналов) | RAK7268 V2 WisGate Edge Lite 2 | Ethernet | 18 000 |
| **Zigbee-координатор** | USB-адаптер Zigbee 3.0 | Sonoff ZBDongle-E (EFR32MG21) | USB | 2 500 |
| **ИТОГО IoT-сенсоры** | | | | **134 700** |

**Разработчик ПО**: Программный стек для IoT — полностью Open Source:
- **ChirpStack** (chirpstack.io) — LoRaWAN Network Server. Управляет всеми LoRa-устройствами. Self-hosted, бесплатно.
- **Mosquitto** — MQTT-брокер (протокол передачи данных IoT). Лёгкий, стабильный.
- **Zigbee2MQTT** — мост между Zigbee-устройствами и MQTT.
- **Home Assistant** (home-assistant.io) — дашборд, автоматизация, уведомления. Самый популярный Open Source проект для IoT.
- **InfluxDB** — база данных временных рядов (хранение показаний датчиков с метками времени).
- **Grafana** — визуализация данных: графики температуры, влажности, энергопотребления, уровня воды.

**Консультант по цифровизации**: Практическая польза IoT для поселения:
- **Метеостанция** → автоматический прогноз заморозков, оповещение в мессенджер «Завтра ночью −3°C, укройте рассаду»
- **Влажность почвы** → агроном видит на телефоне, нужен ли полив, вместо обхода полей
- **Уровень воды** → дежурный видит уровень в накопительных баках, автоматический запуск насоса
- **Энергомониторинг** → понимание расхода энергии по объектам, планирование мощностей
- **Температура теплиц** → уведомление при снижении ниже критической отметки

**Специалист по информационной безопасности**: IoT-устройства — слабое звено безопасности. Обязательные правила:
- Все IoT-устройства — в отдельной VLAN (10.0.150.0/24), изолированной от серверной и пользовательской сети
- IoT-устройства не имеют доступа в интернет (только локальная сеть)
- LoRa-трафик шифруется на уровне протокола LoRaWAN (AES-128)
- Пароли по умолчанию на всех устройствах — сменить сразу при установке

### Итерация 26: VoIP-телефония для внутренней связи

### Итерация 27: Резервное электропитание IT-инфраструктуры

**Системный администратор**: Настройка автоматического поведения при отключении питания:
1. **0–5 секунд**: ИБП принимает нагрузку, серверы работают без перерыва
2. **5 минут**: если питание не восстановлено — автоматический запуск генератора (ATS)
3. **10 минут**: генератор вышел на режим, ATS переключает нагрузку на генератор, ИБП заряжается
4. **30 минут без генератора** (если генератор не запустился): корректное выключение некритичных сервисов (Jellyfin, Grafana)
5. **45 минут без генератора**: корректное выключение всех серверов (скрипт shutdown через NUT — Network UPS Tools)
6. При восстановлении питания: автоматический запуск серверов (Wake-on-LAN или BIOS auto-power-on)

**Специалист по информационной безопасности**: Network UPS Tools (NUT) — Open Source система мониторинга ИБП. Устанавливается на сервер, отслеживает состояние ИБП по USB/SNMP, автоматически выключает серверы при критическом разряде. Обязательный компонент. Уведомления о событиях (отключение сети, переход на генератор, низкий заряд) — в мессенджер через webhook.

### Итерация 28: Стратегия резервного копирования — расширенная версия

**Специалист по информационной безопасности**: В итерации 17 мы описали базовую стратегию 3-2-1. Теперь — детальная реализация с конкретными инструментами и скриптами.

**Инструменты резервного копирования:**

| Задача | Инструмент | Почему |
|--------|-----------|--------|
| Инкрементальный бэкап файлов | **BorgBackup** (borgbackup.org) | Дедупликация (экономия 60–80% места), шифрование, сжатие |
| Бэкап БД PostgreSQL | **pg_dump + pg_basebackup** | Нативные инструменты PostgreSQL, надёжность |
| Бэкап БД MariaDB (ERPNext) | **mariabackup** | Горячий бэкап без остановки сервиса |
| Синхронизация на NAS | **rsync** | Стандарт Linux, инкрементальная синхронизация |
| Офсайт-бэкап (при наличии интернета) | **rclone** | Шифрованная синхронизация на облако (Yandex Disk / S3) |
| Мониторинг бэкапов | **Healthchecks.io** (self-hosted) | Уведомление, если бэкап не выполнился |

**Расписание резервного копирования:**

| Что | Частота | Время | Инструмент | Куда | Хранение |
|-----|---------|-------|-----------|------|----------|
| БД MariaDB (ERPNext) | Ежедневно | 02:00 | mariabackup | NAS:/backups/mariadb/ | 30 дней |
| БД PostgreSQL (Nextcloud) | Ежедневно | 02:15 | pg_dump -Fc | NAS:/backups/postgres/ | 30 дней |
| Файлы Nextcloud | Ежедневно | 03:00 | BorgBackup | NAS:/backups/borg/ | 90 дней |
| Docker volumes (все сервисы) | Ежедневно | 03:30 | BorgBackup | NAS:/backups/borg/ | 90 дней |
| Конфигурация MikroTik | Еженедельно | Вс 04:00 | /export file= + SCP | NAS:/backups/network/ | 52 недели |
| Конфигурация серверов | Еженедельно | Вс 04:00 | BorgBackup /etc/ | NAS:/backups/borg/ | 52 недели |
| Полный образ системы | Ежемесячно | 1-е число, 04:00 | Clonezilla (ручной) | Внешний HDD #1 | 6 месяцев |
| Критические данные → офсайт | Еженедельно | Вс 05:00 | rclone (шифрованный) | Yandex Disk (10 ГБ бесплатно) | 12 недель |
| Внешний HDD → вывоз из поселения | Ежемесячно | Ручной | Borg → внешний HDD #2 | Хранится в городе | 12 месяцев |

**Системный администратор**: Пример cron-задач (добавляются через `crontab -e` на сервере):

```cron
# Бэкап MariaDB (ERPNext) — ежедневно в 02:00
0 2 * * * /opt/scripts/backup-mariadb.sh >> /var/log/backups/mariadb.log 2>&1

# Бэкап PostgreSQL — ежедневно в 02:15
15 2 * * * /opt/scripts/backup-postgres.sh >> /var/log/backups/postgres.log 2>&1

# BorgBackup файлов — ежедневно в 03:00
0 3 * * * /opt/scripts/backup-borg.sh >> /var/log/backups/borg.log 2>&1

# Конфигурация MikroTik — воскресенье в 04:00
0 4 * * 0 /opt/scripts/backup-mikrotik.sh >> /var/log/backups/mikrotik.log 2>&1

# Проверка здоровья бэкапов — ежедневно в 06:00
0 6 * * * /opt/scripts/check-backups.sh >> /var/log/backups/health.log 2>&1
```

**Специалист по информационной безопасности**: Критически важные правила:
1. **Тестируйте восстановление** — раз в квартал делать пробное восстановление из бэкапа на тестовый сервер. Бэкап, из которого нельзя восстановиться — не бэкап.
2. **Шифруйте офсайт-бэкапы** — BorgBackup шифрует по умолчанию (AES-256-CTR + HMAC-SHA256). Для rclone — обязательно включить `crypt` remote.
3. **Мониторинг** — если скрипт бэкапа не отработал, дежурный должен узнать об этом утром. Healthchecks.io (self-hosted) или простой скрипт с отправкой сообщения в Rocket.Chat.
4. **Внешний HDD** — два диска в ротации. Один лежит в серверной (свежий бэкап), второй — в городе (предыдущий бэкап). Раз в месяц меняются местами.

**Стоимость системы бэкапов:**
- NAS Synology DS920+ + 4×4 ТБ: уже учтён в итерации 15 (~100 000 руб)
- Внешние HDD 4 ТБ × 2 шт (Seagate Expansion): 2 × 8 000 = 16 000 руб
- ПО (BorgBackup, rsync, rclone, Healthchecks): 0 руб (Open Source)
- Yandex Disk (10 ГБ бесплатно, 1 ТБ — 300 руб/мес): 0–300 руб/мес
- **ИТОГО доп. расходы на бэкапы**: 16 000 руб разово + 0–300 руб/мес

### 4.2 Модули ERPNext для поселения

ERPNext (https://erpnext.com) — свободная ERP-система на Python/JavaScript (фреймворк Frappe). Самостоятельная установка на сервер, бесплатно.

**Используемые модули:**

| Модуль ERPNext | Применение в поселении |
|----------------|----------------------|
| Accounting (Бухгалтерия) | Бюджет, доходы/расходы, взносы жителей |
| HR & Payroll (Кадры) | Реестр жителей-работников, учёт трудовых часов |
| Stock (Склад) | Учёт материалов, инструмента, запчастей, семян, кормов |
| Manufacturing (Производство) | Домокомплекты, пиломатериал, бетон — BOM, наряды, выпуск |
| Agriculture (Сельское хозяйство) | Карта полей, посадки, урожай, поголовье |
| Buying (Закупки) | Заказы поставщикам, учёт поступлений |
| Selling (Продажи) | Продажа излишков, домокомплектов |
| Project (Проекты) | Строительные проекты, задачи, сроки, исполнители |
| Asset (Основные средства) | Учёт оборудования, станков, транспорта |

**Кастомизация ERPNext:**
ERPNext позволяет создавать пользовательские DocTypes (типы документов). Для поселения нужно создать:
- Тип «Житель» (расширенный профиль: семья, дом, участок, навыки)
- Тип «Дом» (адрес, тип, площадь, состояние, инженерные подключения)
- Тип «Участок» (номер, площадь, назначение, владелец/пользователь)
- Тип «Дежурство» (тип, дата, смена, ответственный)
- Тип «Животное» (вид, кличка/номер, возраст, вакцинации, продуктивность)

### 4.3 Nextcloud — файловый сервер

Nextcloud (https://nextcloud.com) — self-hosted аналог Google Drive / Dropbox.

**Функциональность:**
- Хранение и синхронизация файлов (документы, чертежи, фото)
- Онлайн-редактирование документов (Collabora Online или OnlyOffice)
- Календарь (CalDAV) — общий календарь событий поселения
- Контакты (CardDAV) — общая адресная книга
- Мобильные приложения (iOS, Android)
- Плагины: Talk (видеозвонки), Forms (формы/опросы), Deck (канбан-доска)

**Системные требования:**
- 2+ CPU cores, 4+ ГБ RAM
- 100+ ГБ SSD для ОС + ПО
- HDD массив для файлов (зависит от объёма: от 1 ТБ)
- PHP 8.x, PostgreSQL или MariaDB, Nginx

### 4.4 Мессенджер (Rocket.Chat)

Rocket.Chat (https://rocket.chat) — self-hosted мессенджер, аналог Slack.

**Зачем:**
- Оперативная координация (быстрее email, не требует радио)
- Каналы по подразделениям (#строительство, #сельхоз, #администрация, #общий)
- Файлообмен (фото со стройки, документы)
- Видеозвонки (совещания)
- Push-уведомления на смартфоны (через внутреннюю сеть)

## 5. 15.4 ДОКУМЕНТООБОРОТ И БИБЛИОТЕКА ШАБЛОНОВ

**Основа**: Nextcloud + встроенный модуль workflow.

**Жизненный цикл документа:**
1. **Создание** — из шаблона или с нуля

## 8. КИБЕРБЕЗОПАСНОСТЬ И РЕЗЕРВНОЕ КОПИРОВАНИЕ

### 8.1 Политика безопасности

| Правило | Описание |
|---------|---------|
| Аутентификация | Все пользователи через Keycloak (SSO). Минимум 12 символов пароль. |
| Авторизация | Роли: администратор, бухгалтер, бригадир, житель, гость. Минимальные привилегии. |
| Шифрование | LUKS для дисков сервера. TLS для веб-сервисов. WireGuard для VPN. |
| Обновления | Ежемесячная установка обновлений безопасности (apt upgrade). |
| Firewall | MikroTik firewall на периметре. UFW на серверах. |
| Логирование | Все действия логируются. Логи хранятся 90 дней. |
| Физическая безопасность | Серверная под замком. Журнал доступа. |

### 8.2 Стратегия резервного копирования

**Правило 3-2-1:**
- **3** копии данных (оригинал + 2 бэкапа)
- **2** разных носителя (SSD сервера + HDD NAS + внешний HDD)
- **1** копия за пределами поселения (внешний HDD в городской квартире/у родственников)

| Тип бэкапа | Частота | Носитель | Хранение | Автоматизация |
|------------|---------|---------|----------|---------------|
| Инкрементальный | Ежедневно, 03:00 | NAS (Synology) | 30 дней | Cron + rsync / Borg |
| Полный | Еженедельно, вс 03:00 | NAS (Synology) | 12 недель | Cron + Borg |
| Полный офлайн | Ежемесячно | Внешний HDD (2 шт, ротация) | 12 месяцев | Ручной запуск скрипта |
| Критические данные | Еженедельно | Внешний HDD → увозится из поселения | 6 месяцев | Ручной |
| БД (PostgreSQL) | Ежедневно, 02:00 | NAS + внешний HDD | 30 дней | pg_dump + cron |

**Что бэкапится:**
- База данных PostgreSQL (ERPNext, Nextcloud, Rocket.Chat)
- Файловое хранилище Nextcloud (все документы)
- Конфигурация серверов и сетевого оборудования
- Конфигурация Docker-контейнеров (docker-compose.yml + env)

**Что НЕ бэкапится (восстанавливается установкой):**
- Операционная система
- Приложения (ERPNext, Nextcloud, etc.)
- Кэш

### 8.3 Процедура восстановления

**Документ «Восстановление сервера» — ОБЯЗАТЕЛЬНО распечатать и хранить в серверной!**

Пошаговая инструкция:
1. Установка Ubuntu Server на новое оборудование
2. Установка Docker + Docker Compose
3. Восстановление конфигурации из бэкапа
4. Восстановление базы данных из дампа (pg_restore)
5. Восстановление файлов Nextcloud из бэкапа
6. Проверка работоспособности всех сервисов
7. Перенастройка DNS / IP (если изменилось оборудование)

**Время восстановления**: 2–4 часа при наличии инструкции и бэкапа.

### 8.4 Защита радиосвязи

- DMR-рации с шифрованием (AES-256 или Basic Privacy)
- Смена ключей шифрования каждые 6 месяцев
- Программирование раций только через один выделенный компьютер
- Запасной файл конфигурации раций — на NAS

## ПРИЛОЖЕНИЕ Б: Ключевые решения и обоснования

| Решение | Почему | Альтернативы рассмотрены |
|---------|--------|------------------------|
| MikroTik для сетевого оборудования | Лучшее соотношение цена/функциональность, доступность в РФ, единая ОС | Ubiquiti (дороже, хуже с доступностью), TP-Link (слабее функционально) |
| ERPNext для учёта | Open Source, полнофункциональная ERP, Python/JS, активное сообщество | 1С (платная лицензия, Windows-зависимость), Odoo (частично платная) |
| Nextcloud для документов | Open Source, зрелый проект, мобильные приложения, онлайн-редактор | Google Workspace (облако, зависимость), Seafile (менее функционален) |
| DMR для раций | Цифровое качество, шифрование, 2 канала на частоте, открытый стандарт | Аналоговые (нет шифрования, хуже качество), dPMR (менее распространён), TETRA (очень дорого) |
| Linux для серверов и рабочих станций | Бесплатно, безопасно, стабильно, нет лицензионных рисков | Windows (платные лицензии, вирусы, зависимость от Microsoft) |
| Baofeng для массовых раций | Низкая цена при адекватном качестве, DMR-совместимость | Kenwood (2-3x дороже), Motorola (3-5x дороже) |
| PostgreSQL для БД | Надёжность, производительность, совместимость с ERPNext/Nextcloud | MySQL/MariaDB (менее функциональна), SQLite (не для многопользовательского) |
